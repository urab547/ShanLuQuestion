本文档是写给B（系统策划与前端开发）的专属设计文档，目的是让B在写任何一行代码之前，先有一份完整、可执行、无歧义的规则说明。

文档分为两大部分：

▸ 第一部分（第一章至第五章）：具体玩法系统——本游戏已经设计好的每一个谜题、每一套机制的详细规则、数据结构和触发条件

▸ 第二部分（第六章至第十一章）：基础系统框架——任何一款完整游戏都需要的底层系统，包括存档、引导、反馈、菜单、难度曲线、技术架构

开发原则：每个系统在写代码前，先产出一页纸的规则说明，与A（剧情）和C（美术）确认无误后再动手实现，避免返工。



 

# 第一部分　具体玩法系统

 

 

# 第一章　场景与交互系统

 

 

## 1.1 场景节点结构

游戏世界由若干「场景节点」组成，玩家在节点间自由移动（软性开放，见1.4）。

 

场景节点清单：

village_entrance   村口·玛尼堆

village_square    村子中央广场

dangzeng_courtyard   旦增家院子

dangzeng_bedroom    旦增卧室（后期解锁）

gesang_corner     格桑的角落

temple_exterior    经堂外

temple_interior    经堂内部（第四章解锁）

mountain_peak      山顶·经幡

sangtai        煨桑台

 

## 1.2 场景数据结构

每个场景节点需要包含以下字段，建议以JSON对象数组管理：

{

 sceneId: "dangzeng_courtyard",

 background: "图片资源路径",

 visibleFrom: ["village_square"],   // 从哪些场景可进入

 exits: [{target:"village_square", position:{x,y}}],

 interactables: [物件ID数组],

 npcsPresent: [NPC ID数组，含条件],

 stateVariants: {            // 同一场景不同进度下的差异

   chapter1: {...},

   chapter3: {...}  // 例：藏獒位置、光照变化

 }

}

 

## 1.3 可交互物件系统

每个可点击物件需要统一的数据结构，方便C后期批量替换美术资源而不影响逻辑：

{

 itemId: "mani_stack_note",

 sceneId: "village_entrance",

 hitArea: {x, y, width, height},    // 点击热区

 icon: "hover时是否高亮提示",

 unlockCondition: null,         // 该物件何时可见/可点

 onInteract: "triggerEvent ID",

 oneTimeOnly: true,           // 是否只能触发一次

 resultType: "dialogue" | "item" | "scene_change"

}

 

## 1.4 软性开放世界规则（核心机制）

*场景物理上全部开放，但内容根据玩家进度动态变化。玩家始终能走到任何地方，但「能看到什么、能做什么」由进度决定。*

实现方式：每个场景的interactables和npcsPresent列表，根据全局进度变量（见第七章状态管理）动态过滤。

 

**示例：经堂外场景在不同进度下的差异**

| **进度阶段** | **可见物件**                   | **NPC****反应**            |
| ------------ | ------------------------------ | -------------------------- |
| 第一章       | 锁着的门（可点击但只触发对话） | 白玛："这扇门关了很久了"   |
| 第三章       | 门+新增的玛尼石刻痕            | 卓玛开始愿意谈论钥匙的事   |
| 第四章       | 门自动变为可进入状态           | 无需额外点击确认，直接转场 |

技术要点：不要用「锁定/未锁定」这种二元状态去做，而是用「内容图层」叠加——同一个物件在不同进度下，onInteract指向不同的对话或反应内容，物件本身始终可点击。这样玩家永远不会遇到「灰色不可点」的挫败感。



 

# 第二章　NPC信任与状态系统

 

 

## 2.1 设计原则

总GDD中明确要求：「没有可见的信任值数字，信任通过NPC行为变化体现」。这对B提出了具体的技术要求——内部必须有数值，但UI层绝不能显示数字或进度条。

 

## 2.2 信任值数据结构

{

 npcId: "danzeng",

 trustLevel: 0,      // 内部数值，0-100，玩家不可见

 trustStage: "stranger",  // 映射出的状态标签，决定NPC行为

 flags: {         // 关键事件触发标记，布尔值

   hasDiscussedPainting: false,

   hasSeenBrokenCloth: false,

   hasGivenThangka: false

 },

 unlockedDialogueTopics: [] // 随信任值解锁的可聊话题

}

 

## 2.3 信任阶段映射表

每个NPC需要策划单独定义阶段映射，以旦增为例：

| **数值区间** | **阶段标签** | **行为表现**               | **解锁内容**   |
| ------------ | ------------ | -------------------------- | -------------- |
| 0–20         | stranger     | 只回答"不知道"或沉默       | 无             |
| 21–45        | listener     | 愿意聊绘画、聊山，不聊唐卡 | 残卷第四页线索 |
| 46–70        | acknowledged | 主动提起祖父，认可玩家     | 深层对话选项   |
| 71–100       | confidant    | 主动出现，说出真相         | 经堂剧情触发   |

 

## 2.4 信任值增长规则

信任值不应该靠「对话次数」简单累加，而要靠「对话质量」——这是策划最需要明确定义的部分，否则A写对话、B做系统时双方标准会不一致。

 

**增长触发器设计**

| **话题契合 +3~5**      | 玩家主动聊NPC感兴趣的话题（如对旦增聊绘画），由AI返回的对话打上语义标签触发 |
| ---------------------- | ------------------------------------------------------------ |
| **解谜成果展示 +8~15** | 玩家完成某个谜题后向NPC展示成果（如赠予颜料知识、唐卡复刻）  |
| **文化理解正确 +5**    | 玩家在AI对话中表现出对藏族文化的准确理解（需AI判断后返回标记） |
| **触碰禁忌话题 -5~10** | 在信任不足时直接追问核心秘密，AI识别后返回负面标记           |
| **长时间不互动**       | 不衰减——这款游戏没有"好感度会掉"的设计，信任只增不减，保护玩家探索的安全感 |

 

## 2.5 AI返回标记与信任系统的对接方式

这是B需要和A共同设计的接口规范。AI对话每次返回时，除了对话文本本身，还需要返回一个结构化的标记对象，供前端更新信任值：

{

 dialogueText: "你祖父当年也是这样问我的...",

 trustDelta: +8,

 triggeredFlags: ["hasDiscussedPainting"],

 unlockedTopics: ["祖父的田野调查"]

}

实现建议：在每个NPC的System Prompt末尾追加格式要求，让AI在对话正文后附加这样一段JSON，前端解析后更新本地状态，正文部分展示给玩家，JSON部分不展示。



 

# 第三章　解谜系统详细设计

 

 

## 3.1 残卷收集系统

 

**数据结构**

{

 scrollPageId: "scroll_page_1",

 location: "village_entrance.mani_stack",

 unlockCondition: "always_visible_but_findable", // 第一页无前置条件

 contentRevealed: "构图基础比例规则",

 collected: false

}

 

**四页残卷触发条件一览**

| **残卷** | **位置**     | **解锁条件**                     | **内容**         |
| -------- | ------------ | -------------------------------- | ---------------- |
| 第一页   | 玛尼堆石缝   | 无条件，仔细检视即得             | 构图比例基础     |
| 第二页   | 白玛念经角落 | 白玛信任值≥21（listener阶段）    | 佛青颜料隐喻线索 |
| 第三页   | 格桑画夹     | 格桑主动展示画夹的对话分支触发后 | 藏金颜料制法线索 |
| 第四页   | 旦增院子石下 | 旦增信任值≥21且已完成"谈画"对话  | 骨白颜料情感线索 |

策划要点：残卷不是按顺序解锁的线性流程，而是和四个不同NPC的信任值挂钩，玩家可能先认识格桑、后认识旦增，导致拿到残卷的顺序因人而异——这是有意为之的设计，不需要强制顺序，但B在实现时要确保任意顺序组合都不会导致逻辑冲突（例如构图谜题界面要能正确处理"残卷不全也可以先进入界面看已解锁部分"的情况）。

 

## 3.2 颜料采集与制作系统

 

**五种颜料数据结构**

{

 pigmentId: "fojing_blue",

 name: "佛青",

 rawMaterialFound: false,  // 是否已采集原材料

 recipeUnderstood: false,  // 是否已理解制法

 craftable: false,      // 两者皆true时变为true

 craftedCount: 0

}

 

**原材料采集谜题——观察判断型**

以佛青矿石为例的具体实现逻辑：

▸ 场景中放置两个视觉相似的蓝色矿石物件：blue_stone_correct 和 blue_stone_decoy

▸ 两者初始外观相同，但定义一个"光照角度"互动——玩家需要点击场景中的光源切换按钮（或拖动视角滑块），在特定角度下，正确的矿石会显现出"深秋夜空"般的渐变色，错误的保持纯色

▸ 点击正确矿石触发采集成功；点击错误矿石不会有任何惩罚，只返回NPC旁白提示「这个颜色不太对」，鼓励玩家继续尝试

这是典型的"观察谜题"模板，B可以把这个模板复用在松石绿等其他需要场景观察的颜料上，只需替换判断条件和视觉资源。

 

**颜料制法解锁——对话/场景信息型**

五种颜料的制法解锁渠道不同，B需要分别对接：

| **颜料** | **制法来源类型** | **触发方式**                              | **技术对接点**                                               |
| -------- | ---------------- | ----------------------------------------- | ------------------------------------------------------------ |
| 佛青     | AI对话隐喻       | 白玛对话中AI返回特定flag                  | 监听triggeredFlags中是否含recipe_fojing                      |
| 朱砂红   | 场景细节         | 点击院子铜炉，刻字仅在特定光照/缩放下可读 | 场景物件的多阶段交互（先点击→显示提示→需要"擦拭"动作→显示刻字） |
| 藏金     | 对话+残卷联动    | 格桑对话提及植物，需与已收集残卷比对      | 前端需做简单的"比对"小游戏或自动判断已有信息是否匹配         |
| 松石绿   | 场景物件         | 点击老药方物件，对照已知植物特征          | 文本比对型谜题，可做简单的"选择正确植物"交互                 |

 

**骨白颜料——核心情感谜题**

这是全游戏唯一不能仅靠"采集+制法"两步完成的颜料，需要单独设计触发链：

**1.**　前置条件：玩家已通过对话理解骨白与旦增亡妻的关联（AI对话触发特定flag: understood_bone_white_meaning）

**2.**　玩家向旦增"追问"骨白制法（需专门设计一个对话入口，不是自由聊天，而是在特定信任阶段出现的固定对话选项）

**3.**　旦增交付一小块氆氇残片（白玛保存的同款碎片）作为道具，前端在道具栏新增bone_white_cloth物件

**4.**　玩家在颜料制作界面"研磨"该物件，触发骨白解锁动画与旁白

技术要点：这一段不能用前面四种颜料的通用模板，需要单独写一段线性引导逻辑，因为它是情感核心，必须保证每个玩家都不会错过这段关键叙事。



 

## 3.3 唐卡制作界面（核心创作系统）

 

**界面构成**

制作界面由三层叠加的子谜题组成，建议做成分步骤的Modal界面，而非单一复杂画布：

 

### Step 1：构图谜题

网格化拖拽：界面提供一个比例网格（来自C的UI素材，第八章已定义），玩家需要把度母的身体部位元素（头部、躯干、手势、莲花座等预制图形）拖拽到符合"藏族量度规范"比例的网格位置。

gridRules: {

 headPosition: {row: 1-2, col: 4-6},   // 正确范围

 torsoPosition: {row: 3-6, col: 3-7},

 tolerance: 1  // 允许的格数误差，避免过度刁难

}

判定逻辑：每个元素拖放后，比对其网格坐标是否落在rule容差范围内，全部正确才能进入下一步；错误时元素自动弹回原位，并显示一句来自残卷的提示文本（不是错误音效，避免挫败感）。

 

### Step 2：颜色谜题

玩家从已制作好的颜料中，为构图的不同区域（护法部分、衣纹部分、背景部分等）分配颜色。

错误示例（按GDD设定）：把护法部分涂成蓝色是文化错误。判定后不是直接"错误"提示，而是触发白玛的旁白解释为什么——这是把谜题反馈转化为文化学习的关键设计点，B需要为每个可能的错误分配预设的解释文本ID，从A处获取文案。

 

### Step 3：细节谜题——度母左眼

这一步不是操作型谜题，而是选择型——界面呈现几种"心中所念"的隐喻选项（不直接写"妻子"），玩家需要根据此前所有对话积累的理解，选择最贴近的一项。

技术实现：这是少数几个允许"选错"的环节，选错不会卡住流程，但旦增后续的反应文本会有细微差异（通过flag区分），鼓励玩家细心程度的同时不惩罚探索。

 

**唐卡制作完成后的产出物**

制作完成后，系统生成一个唯一的thangka_replica对象，记录玩家在三个步骤中的具体选择（用于后续赠予分叉判断玩家的理解程度，但不影响是否能赠予——赠予本身永远可行，差异体现在NPC反应文本上）。



 

## 3.4 赠予分叉系统

这是重玩价值的核心，B需要设计一套通用的"赠予"交互框架，而不是为三个NPC各写一套独立逻辑。

 

**通用赠予交互流程**

**1.**　玩家在道具栏选中thangka_replica，点击任意已解锁的可赠予NPC

**2.**　系统检查该NPC是否已被赠予过（giftedNPCs数组判重，防止重复触发同一段关键剧情）

**3.**　触发该NPC专属的赠予反应对话（固定剧情，非AI自由生成——这是关键节点，必须保证叙事准确，不能交给AI即兴发挥）

**4.**　赠予后更新对应flag（如triggered_danzeng_box_reveal），解锁后续场景或对话分支

重要设计澄清：赠予动作本身可以重复执行给不同NPC（旦增、格桑、白玛都可以分别赠予一次），系统需要支持"一份道具，多次使用"，而不是消耗品逻辑。



 

## 3.5 图像解读系统（唐卡谜题层）

五个区域的解读各自独立，建议做成"热点拼图"形式：玩家收集到唐卡局部照片碎片后，在一张组合界面上将其拼合，每个区域被点击时触发对应的解读对话或旁白。

 

| **区域**        | **解锁前置**               | **交互方式**                                   | **结果**               |
| --------------- | -------------------------- | ---------------------------------------------- | ---------------------- |
| 区域一·手印     | 获得仁青提供的对应碎片     | 点击后弹出解读文本+与旦增行为的呼应提示        | 文化知识+人物呼应      |
| 区域二·背景山形 | 场景对比                   | 并排显示场景实拍与唐卡局部，玩家手动确认"相似" | 纯视觉发现，无对错判定 |
| 区域三·度母左眼 | 积累信息量达阈值           | 自动解锁，不需操作                             | 核心象征揭示，文本展示 |
| 区域四·角落藏文 | 请白玛辨认（专属对话触发） | 固定剧情对话，非自由聊天                       | 名字关联揭示           |
| 区域五·画面空白 | 完成其余四个区域           | 自动触发                                       | 故事核心隐喻收束       |

技术要点：这五个区域不要求玩家按顺序完成，但区域五需要前四个全部完成才能触发——这是唯一的强制顺序点，需要在代码里做完成度检查。



 

## 3.6 仪式重现系统（煨桑台）

 

**系统结构**

四个子条件需要同时满足，任一错误都会触发对应的失败反馈（非惩罚性，是引导性的）：

| **条件** | **正确选项来源**                                             | **错误反馈**                     | **数据结构字段**               |
| -------- | ------------------------------------------------------------ | -------------------------------- | ------------------------------ |
| 植物种类 | 场景采集，松柏/杜松等可选项                                  | 白玛皱眉不语（无文本，纯演出）   | plantType: correct/incorrect   |
| 燃烧顺序 | 残卷或白玛对话提及"先净后请"                                 | 烟雾视觉效果显示往下走           | burnOrder: array判断顺序       |
| 诵念内容 | 此前白玛闲聊时出现过的词句，需玩家记住或游戏内有"回顾对话"功能可查 | 白玛停下纠正，触发额外剧情       | chantText: 字符串匹配          |
| 点火时机 | 白玛提到的藏历时辰                                           | 无明确错误反馈，允许玩家随时尝试 | timing: 非强制判定项，仅做氛围 |

特别说明：点火时机这一项不建议做成强制判定——藏历时辰的实时换算对开发成本过高，且容易显得刻板。建议将其设计为"氛围彩蛋"：如果玩家凑巧在游戏内时间的对应时段点火，触发一句特别的旁白，但不影响仪式成败判定。

 

**仪式成功演出**

全部条件满足后，触发一段不可跳过的演出序列：烟雾笔直上升的动画→风声渐弱→村民NPC陆续从各自场景"走入"煨桑台场景（需要B实现简单的NPC位置切换逻辑，让NPC图层临时出现在这个场景）→旦增最后出现。

这是全游戏少有的"播片"式时刻，建议用固定时间轴的方式实现（类似简单的时间轴动画序列），而非交互式过程。



 

## 3.7 六字真言拼图

六个石块乱序放置在经堂门口场景，玩家需要拖拽排列成正确顺序（唵嘛呢叭咪吽）。

{

 correctOrder: ["om","ma","ni","pad","me","hum"],

 currentOrder: [玩家当前摆放],

 hintSource: "白玛此前对话中念出过完整顺序",

 checkOnEachMove: true  // 每次拖拽后即时判断，不需要"提交"按钮

}

设计意图提醒（来自总GDD）：这个谜题考验的是玩家有没有认真听白玛说话，不是记忆力游戏。建议在游戏内提供一个轻量的"对话回顾"入口（见第六章），让错过台词的玩家可以查阅，避免因为没记下来而卡关太久。

 

## 3.8 经幡排序玩法

**设计定位**

经幡谜题不是孤立的小游戏，而是白玛专属的剧情台阶，作用是在六字真言拼图之前，给白玛这条线索增加一个递进节点。两个谜题共用同一套拖拽排序模板，但分别挂在白玛信任值的不同阶段，形成"低信任→经幡谜题，高信任→六字真言拼图"的递进关系。



**触发条件**

玩家与白玛的信任值达到 listener 阶段（21–45区间）后，白玛的对话池解锁一句新台词，同时山顶·经幡场景（mountain_peak）中的经幡从纯展示物件变为可交互谜题物件。

提示台词示例（最终文案由A确认）： "风把颜色吹乱了，蓝天白云火焰绿水大地，谁还记得它们该在哪里。"

这句话本身承担谜题提示的作用，呼应六字真言拼图"考验玩家有没有认真听"的设计逻辑，不需要额外的UI教程提示。



**谜题机制**

**数据结构**

{

 correctOrder: ["blue", "white", "red", "green", "yellow"],

 currentOrder: [玩家当前排列],

 unlockCondition: "baima.trustStage >= listener",

 checkOnEachMove: true,

 solved: false

}

**交互方式**

拖拽排序，五条经幡按蓝、白、红、绿、黄从上到下重新排列。每次拖拽后即时判断，不需要"提交"按钮，与六字真言拼图保持一致的交互手感。

**技术实现**

完全复用 3.7 六字真言拼图的拖拽排序组件，仅替换数据内容（correctOrder 数组和对应的视觉资源），不需要新写交互逻辑。建议将拖拽排序逻辑抽象为通用组件（如 DragSortPuzzle），供两处谜题共同调用。



**完成反馈**

经幡排列正确后，不使用第八章定义的通用"正确反馈组件"（金色光效+钵声）直接收尾，而是触发白玛的专属对话，将经幡的"颜色归位"和经堂的"门关闭"做一次隐喻类比：

"色归其位，心才安。这经堂的门，比这经幡乱得更久。"

这句话的作用是为玩家埋下关于经堂关闭的新暗示，不直接给出信息，保持白玛一贯的双层语言风格。



**信任值联动**

谜题完成属于"解谜成果展示"类触发器，按第二章2.4定义的区间给予白玛信任值 +8~15。

具体数值建议取中段（+10），与之前已验证的旦增、卓玛测试面板的+10按钮量级保持一致，便于测试阶段直接复用同一套验证方式。



**与六字真言拼图的关系**

| **项目**     | **经幡排序**            | **六字真言拼图**             |
| ------------ | ----------------------- | ---------------------------- |
| 触发信任阶段 | listener（21分以上）    | 不限定，固定出现在经堂门口   |
| 场景位置     | 山顶·经幡               | 经堂外门口                   |
| 提示来源     | 白玛新增台词            | 白玛此前对话中念出的完整顺序 |
| 完成后效果   | 信任值+10，解锁隐喻台词 | 解锁进入经堂的最后一道门槛   |
| 谜题模板     | DragSortPuzzle（复用）  | DragSortPuzzle（原始实现）   |

两个谜题在结构上是同一套机制的两次复用，但叙事功能不同：经幡谜题是白玛线索的"前台阶"，六字真言拼图是"最后一道门槛"。开发时建议先完成经幡排序，验证 DragSortPuzzle 组件的复用性，再视情况决定是否需要把六字真言拼图也重构为同一个通用组件。

 

 



 

# 第四章　卡关提示机制设计

 

 

## 4.1 设计原则

*提示给的不是答案，而是文化语境的线索。提示同时推进文化理解和人物关系，不能是纯粹的"功能性攻略提示"。*

 

## 4.2 卡关判定规则

B需要为每个谜题定义"卡关阈值"，建议采用时间+尝试次数双重判定，避免单一维度误判：

| **时间阈值** | 玩家在同一谜题界面停留超过3分钟未推进                        |
| ------------ | ------------------------------------------------------------ |
| **尝试阈值** | 玩家在同一交互点重复触发3次以上仍未成功                      |
| **触发条件** | 两者满足其一即视为"可能卡关"，而非强制弹出提示，而是在场景中让相关NPC的可点击状态出现细微高光提示，鼓励玩家主动去问 |

重要原则：不要做成自动弹窗式提示。这款游戏的氛围要求玩家始终感觉「自己在探索」，弹窗会打破沉浸感。正确做法是让提示的获取也需要玩家主动行为（点击NPC对话），只是降低这次对话触发提示的概率门槛。

 

## 4.3 各谜题对应的提示设计

| **卡关谜题** | **触发NPC**  | **提示文本方向**                                             | **技术实现**                             |
| ------------ | ------------ | ------------------------------------------------------------ | ---------------------------------------- |
| 骨白材料     | 白玛         | 闲聊中提到"旦增老伴走的时候，他一件衣服都没留，只留了一块布" | 在白玛对话池中加入低权重但条件触发的台词 |
| 六字真言顺序 | 格桑         | "白玛每天念的那几个字，我从小就会背"并直接念出               | 格桑对话分支，触发后直接显示正确顺序文本 |
| 佛青矿石辨认 | 卓玛或场景内 | 提示玩家尝试改变光照/视角                                    | UI层提示性动画，非文字                   |
| 构图比例     | 旦增         | 重新强调"量度"概念的对话                                     | 旦增对话池追加                           |



 

# 第五章　解谜难度与节奏曲线

 

 

## 5.1 整体难度递增设计

45分钟的游戏时长内，谜题难度应呈现"低-中-高-低"的曲线，而非持续递增——结局阶段需要情感释放，不适合高难度操作。

 

| **章节** | **时间区间** | **谜题难度**       | **谜题类型**                        |
| -------- | ------------ | ------------------ | ----------------------------------- |
| 第一章   | 0–8分钟      | 极低（无实质谜题） | 纯探索，建立操作习惯                |
| 第二章   | 8–18分钟     | 低                 | 单步骤对话型谜题（谈画破冰）        |
| 第三章   | 18–30分钟    | 中                 | 残卷收集+颜料采集（多步骤但容错高） |
| 第四章   | 30–40分钟    | 中高               | 唐卡制作三重谜题（全游戏操作峰值）  |
| 第五章   | 40–45分钟    | 无                 | 无操作型谜题，纯叙事抉择            |

 

## 5.2 容错设计统一规范

全游戏没有"失败结局"或"游戏结束"状态。所有谜题的错误反馈都应遵循以下统一规范，B在实现任何新谜题时都应对照此规范：

**1.**　错误操作不会导致道具丢失或状态倒退

**2.**　错误反馈优先用演出/旁白形式，其次才是文字提示，最后才考虑明确的"错误"视觉信号（红色高亮等），且应尽量避免使用

**3.**　任何谜题理论上允许无限次尝试

**4.**　错误尝试本身可以成为卡关提示系统的触发条件之一（叠加计数）



 

# 第二部分　基础系统框架

 

 

以下是任何完整游戏都需要的底层系统，目前在总GDD中缺失，需要B单独定义功能范围和实现方式。



 

# 第六章　存档与进度系统

 

 

## 6.1 技术方案：localStorage本地存储

无需后端，玩家关闭浏览器后数据保留在本地。缺点是更换设备或清除浏览器数据会丢失存档，需要在游戏内对此做出说明（见6.4）。

 

## 6.2 需要持久化的数据结构

{

 saveVersion: "1.0",

 currentScene: "village_square",

 playerName: "玩家自定义",

 npcTrust: { danzeng: {...}, baima: {...}, ... },

 globalFlags: { chapter: 3, hasMetGesang: true, ... },

 inventory: ["scroll_page_1", "scroll_page_3", ...],

 pigments: { fojing_blue: {craftable:true}, ... },

 thangkaReplica: {...} | null,

 giftedNPCs: ["danzeng"],

 dialogueHistory: { danzeng: [对话记录数组] }, // 用于AI上下文延续

 lastSaveTimestamp: 1234567890

}

 

## 6.3 保存触发时机

不建议做成"手动存档"模式（这款游戏没有失败重来的需求），应做成自动持续保存：

▸ 每次场景切换时自动保存

▸ 每次对话结束（AI返回一轮完整应答）后自动保存

▸ 每次谜题状态变化（如颜料制作完成）后自动保存

技术实现：用一个全局的saveState()函数，在关键状态变更的地方统一调用，避免分散在各处手写localStorage.setItem。

 

## 6.4 读档与重新开始

玩家重新打开网页时，检测localStorage中是否存在存档：

**1.**　存在 → 直接恢复到上次的场景与状态，无需额外确认弹窗（保持沉浸感）

**2.**　不存在 → 进入新游戏流程

需要在设置菜单中提供"重新开始"选项，点击后需二次确认（这是少数允许使用弹窗确认的场景，因为涉及数据清除），确认后清空localStorage对应字段。



 

# 第七章　引导与新手体验系统

 

 

## 7.1 设计决策：极简引导

总GDD明确要求"无任何教程提示"，但完全没有引导会导致部分玩家不知道如何开始交互。B需要设计一种不破坏氛围的隐性引导。

 

## 7.2 隐性引导方案

| **引导目标**   | **实现方式**                                                 | **触发时机**                                                 | **是否可关闭**             |
| -------------- | ------------------------------------------------------------ | ------------------------------------------------------------ | -------------------------- |
| 物件可点击提示 | 鼠标悬停时轻微的光晕呼吸动画，而非图标或文字                 | 玩家进入新场景后，若30秒内无任何点击行为，第一个关键物件开始极轻微的高光提示 | 可在设置中关闭             |
| 对话框使用方法 | 首次打开AI对话框时，输入框内有极淡的placeholder文字示例，如"问问他关于唐卡的事…" | 仅第一次对话触发，之后不再出现                               | 自动消失，无需手动关闭     |
| 场景切换方式   | 场景边缘的出口区域，鼠标悬停时光标变化+轻微脚步光效          | 常驻，不算作"教程"，属于UI反馈                               | 不可关闭，属于基础交互反馈 |

原则：所有引导必须能被理解为「游戏世界本身的反应」，而不是「系统在教你怎么玩」。光晕、呼吸感、光标变化都符合这个原则；文字气泡教程「点击这里开始」不符合，应避免。



 

# 第八章　反馈与状态提示系统

 

 

## 8.1 统一交互反馈规范

B需要制定一套全局通用的反馈规则，确保玩家点击任何物件都能获得一致的体验逻辑，不会出现"有些物件点了有反应，有些没反应却不知道为什么"的困惑。

 

| **物件状态**    | **视觉表现**                                | **点击结果**                                         | **示例**             |
| --------------- | ------------------------------------------- | ---------------------------------------------------- | -------------------- |
| 可交互·未触发   | 静止状态，hover时轻微高亮                   | 触发对应事件                                         | 残卷藏匿点           |
| 可交互·已触发过 | 视觉上保持原样，但内部标记为已读            | 重复点击显示简短的"已经看过"类回应，而非重复完整剧情 | 已读取的场景细节物件 |
| 暂不可交互      | 物理上仍可点击（软性开放原则）              | 返回该阶段对应的占位反应文本                         | 尚未到时机的经堂门   |
| NPC对话入口     | 立绘呼吸动画，hover时光标变化为对话气泡图标 | 打开AI对话框                                         | 所有NPC              |

 

## 8.2 谜题正确/错误的统一反馈语言

呼应第五章的容错设计规范，B需要实现一套可复用的反馈组件：

▸ 正确反馈组件：轻微的金色光效+短促的钵声音效（前文音效设计中提到的"提示音"，全程不超过5次的预算需要在这里被消耗，需与A、C协调具体使用在哪几个最关键的节点）

▸ 引导型反馈组件：无音效，仅展示一行来自残卷或NPC的旁白文字，样式与对话框统一但出现位置在谜题界面内部

技术建议：把这两种反馈做成两个可复用的React组件（CorrectFeedback / GuidanceFeedback），所有谜题模块统一调用，不要各自为政地写局部反馈逻辑。



 

# 第九章　菜单与设置系统

 

 

## 9.1 设置菜单功能范围

| **音量调节**      | 背景环境音/音效音量独立滑块，无BGM音量项（本游戏无背景音乐，仅两处特殊音乐场景，归类入音效） |
| ----------------- | ------------------------------------------------------------ |
| **重新开始**      | 二次确认后清空存档，回到游戏开场                             |
| **对话历史回顾**  | 见9.2，独立模块说明                                          |
| **关于/制作信息** | 团队名单、致谢藏族文化顾问                                   |

 

## 9.2 对话历史回顾系统（线索回顾）

这是为六字真言拼图等"需要记住NPC说过的话"的谜题专门设计的辅助功能，避免玩家因为没记下台词而过度卡关。

 

**功能设计**

▸ 从主菜单或场景内的固定UI入口（建议放在屏幕角落，不强制使用）进入

▸ 分NPC展示已发生过的关键对话片段，而非完整对话记录——只摘录被标记为"keyLine: true"的台词

▸ 技术实现：在NPC对话返回的结构化数据中（参照第二章2.5的JSON格式），增加一个isKeyLine字段，由A在设计台词时标注哪些是"今后可能需要回顾的关键信息"

 

## 9.3 道具栏与残卷/颜料图鉴

常驻UI，展示已收集的残卷页数（如"2/4"，这个数字提示是允许的，因为它是收集型反馈，不是信任度这种情感数值）、已掌握的颜料种类、当前是否持有唐卡复刻品。

点击任意已收集物品可查看详情卡片（文字+示意图标），不消耗任何点击次数限制。



 

# 第十章　难度与节奏统一控制规范

 

 

## 10.1 为什么需要这一章

总GDD中每个谜题的难度是分别设计的，容易出现"各自为政"的问题——比如某个谜题的策划者觉得自己的谜题应该很难，另一个觉得应该很简单，最终玩家体验到的曲线会很跳跃。本章为B提供统一的全局控制原则，所有具体谜题设计都应回头检查是否符合这里的规范。

 

## 10.2 AI对话自由度边界

这是最容易失控的部分。B需要在前端层面对AI对话设置统一的边界控制，而不是完全依赖System Prompt：

| **单轮对话长度限制** | 前端限制单次玩家输入字符数（建议300字以内），避免对话偏离游戏节奏过远 |
| -------------------- | ------------------------------------------------------------ |
| **话题强制纠偏**     | 若AI返回内容被判定明显偏离游戏世界观（可设置简单的关键词黑名单做基础过滤，复杂语义判断交由System Prompt层处理），前端可拒绝展示并提示玩家"换个问题试试" |
| **对话轮次软上限**   | 单次对话会话超过约15轮无实质推进时，NPC立绘可触发"需要去做点别的事了"类型的收尾对话，温和地引导玩家离开当前对话 |

 

## 10.3 全局节奏检查清单

每新增一个谜题或系统时，B应对照以下清单自查：

**1.**　这个谜题的预期完成时间是多久？是否符合第五章的章节时间预算？

**2.**　错误反馈是否符合第五章的容错统一规范？

**3.**　是否需要接入卡关提示系统？提示内容是否已与A确认？

**4.**　UI呈现是否需要新的美术资源？是否已与C同步需求？

**5.**　该谜题完成后是否正确触发了存档（第六章）？
